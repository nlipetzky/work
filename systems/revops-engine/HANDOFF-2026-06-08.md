## prep_run_status — work spec
**Goal:** watch the play-prep funnel *move* in projection-ui (Runs page), not just see the result.
The data-observability exists; run-observability doesn't.

**DB:** new table `prep_run_status(run_id uuid, batch_id text, entity text, stage text,
status text /* running|done|error */, counts jsonb, message text, started_at timestamptz,
updated_at timestamptz)`. One row per (run_id, stage).

**Runners write status:** each runner in `systems/revops-engine/` (`run-stage1.mjs`,
`classify-runner.mjs`, `dedup-runner.mjs`, `route-runner.mjs`, `contacts-screen-runner.mjs`,
`generate-prep-plan.mjs`) writes a `running` row at start and `done`/`error` (+ counts) at finish,
via the Management API (same pattern they already use). Add a `--run-id` arg; the **planner
sub-agent** generates one run_id and threads it to every runner so the stages group.

**Surface:** a section on the projection-ui **Runs** page (`app/runs/`) that tails
`prep_run_status` for the active/most-recent run and renders a live strip:
`Stage-1 ✓ 5 decided · Classifier 40/77 · Dedup ✓ 4 · Routing ✓ · Contacts ✓ 85 eligible`.
Reuse the existing fetch/poll pattern; a simple 2–3s poll is fine.

**Acceptance:** run the funnel on a batch and watch the Runs page advance stage-by-stage, with the
error case visible if a stage throws.
