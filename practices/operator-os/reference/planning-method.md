# Planning Method — the strict guidance the system enforces

The contract the operator-OS planner runs on **every** intent. It exists because Nick's hardest
executive-function task is planning and articulating what to do. So the rule is inverted from normal
software: **the user brings fuzzy, under-articulated intent; the SYSTEM supplies the structure.** Nick
never has to articulate the shape, sequence it, or repeat the goal. If the structure had to come from
Nick, the system failed.

## The whole model: three modes against systems

Nick's entire workload reduces to three things, always against a **system**:

1. **Run** an existing system.
2. **Iterate** an existing system.
3. **Build** a new system.

(Plus a residue of irreducible ad-hoc human work — a decision, a relationship, an approval. Named as
such, never the default.)

Every item the system surfaces must resolve to one of these three, naming the system. "Stand up a
CIPO website" is not a valid item; "Build the CIPO demand-context system" is.

## What each mode requires

- **Run** — the system must be trustworthy: built properly, with its own surface/UI in the projection
  UI, that Nick can operate and observe. Running = operating a live system on its surface. It is
  ongoing operation, not a project.
- **Iterate** — improving a live system. A project (often small) whose tasks are the changes.
- **Build** — bringing a new system into being. **The build IS a project. The system the project
  builds is the project's whole point. The tasks in the project are just the steps to complete the
  build.** Stated as simply as *"create X system that does Y."* On completion the system is live with
  a surface, and moves into Run/Iterate.

So: **project ≡ building (or iterating) a system.** A project that doesn't build or iterate a system
is a flag. A task is a build step of its project's system.

## What the planner does with any intent (strictly, in order)

1. **Objective.** Name the one real objective. Everything else is secondary and labeled secondary.
2. **Resolve to systems.** Which systems must **run** to achieve this? Which must be **iterated**?
   Which don't exist and must be **built**? Every downstream need maps to a system.
3. **Foundation = the system that comes first.** The foundational dependency is the system whose
   output everything else needs. If it isn't live, the first move is to **build** it. (CIPO: the
   demand-context system that produces offer/copy/TAM/targeting is foundational and not yet built;
   the website is a downstream output of other systems.)
4. **Sequence by dependency.** Order by which system produces whose inputs. A system whose inputs
   don't exist yet is not yet runnable; surfacing its output as the next move is wrong, however
   urgent it looks.
5. **The one first move.** Run / iterate / build the foundational system. Never a downstream
   deliverable; never Nick's manual labor.

## What the surface must present

Not a flat scored task list. Work grouped by mode — **Build / Iterate / Run** — each naming its
system, foundation first, with everything secondary marked secondary. A build project shows its system
and its build steps. The structure reached in conversation must live on the surface.

## The guarantee

Fuzzy intent in → a plan of run/iterate/build moves against named systems, foundation-first, on the
surface. Nick judges and corrects; he never supplies the structure or repeats the goal. The
strictness lives here, in the system — not in Nick having to be strict.
