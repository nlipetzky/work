# Canon System Operating Manual (for Atlas)

This is Atlas's manual for using canon_engine. The Canon System's own definition is in the System Registry (record `recggwUTDke8Y7UMe`) and on the database via `canon_engine._ai_context`. This doc is the operator-os angle: how Atlas reads canon_engine to support Nick, what Atlas may write, and what Atlas must escalate.

System pointer: Supabase project `mzzjvoiwughcnmmqzbxv`.

## What canon_engine is, in your terms

Five things:

1. **Past-session memory.** `agent_sessions` table. One row per Claude Code session at close. The cross-engagement log.
2. **Meeting transcripts.** `transcripts` table. Drive-sourced. Konstellation AI calls only as of 2026-06-01.
3. **Triaged email.** `email_threads` (parent) + `email_messages` (children). LLM-summarized, Eisenhower-classified (urgency, importance, quadrant).
4. **Artifact corpus.** `canon_docs` (155 rows of governance/architecture/protocol docs). `canon_artifacts` schema exists but is empty.
5. **Vector retrieval.** `chunks` table, 6,346 rows across four source_types (canon / email / transcript / document). The semantic search surface.

## What canon_engine is NOT

- Not the Work base. Operator state (Projects, Tasks, Weekly Intent, Notifications) lives in Airtable `appz7I91uNxWBnly8`.
- Not the System Registry. Systems and Assets live in Airtable `apppQjlZiktpbO4aX`.
- Not your in-session memory. Model context handles that within a single session.

## Tables you read

Durable, queryable freely:

- `agent_sessions` — your own log; also other personas' sessions
- `transcripts` — meeting recordings (konstellationai today)
- `email_threads` — triaged inbox parent rows
- `email_messages` — message-level children
- `canon_docs` — artifact body store
- `canon_clusters` — topic clusters (rarely needed)
- `canon_artifacts` — empty today; skip until populated

Derived / pipeline-owned (read OK, never write):

- `chunks` — vector embeddings, regenerated by pipeline
- `canon_transcript_state`, `canon_email_state`, `canon_document_state` — ingestion checkpoints
- `voyage_usage_log`, `pipeline_runs`, `sync_log`, `ingestion_manifest`, `canon_events`, `wake_log`, `paperclip_routing`, `domain_lookup`, `enrichment_configs`, `documents`

## What you may write

- `INSERT INTO agent_sessions` at session close (preferably via harness Stop hook; manual fallback).
- `UPDATE agent_sessions` on your own active session row (incremental updates to summary, decisions, etc.).
- `INSERT INTO capture_items` when explicitly logging a captured item, with `session_id` referencing your active session.

Nothing else. If a use case seems to require more, stop and ask Nick.

## What you must never write

- `DELETE` on `email_threads`, `email_messages`, `transcripts`, `canon_docs`, `canon_artifacts`, `chunks` ... data loss.
- `UPDATE` on `chunks` directly ... regenerate via embedding pipeline.
- `INSERT` or `UPDATE` on `canon_*_state` tables ... pipeline-owned.
- Direct writes that bypass ingestion to derived state.

## Query patterns

### Pattern A: cross-engagement recall

Nick: "Did I talk about X recently?"

```sql
SELECT id, started, title, summary, system_slug, account_name
FROM agent_sessions
WHERE summary ILIKE '%X%'
   OR key_decisions ILIKE '%X%'
   OR topics ILIKE '%X%'
ORDER BY started DESC
LIMIT 10;
```

For fuzzier matches, prefer the semantic `recall` move (Pattern F) once `agent_sessions` are embedded.

### Pattern B: transcript lookup

Nick: "What did Will say about list quality?"

```sql
SELECT transcript_title, meeting_date, summary, key_decisions, action_items
FROM transcripts
WHERE (key_decisions ILIKE '%list quality%'
    OR summary ILIKE '%list quality%'
    OR raw_transcript_text ILIKE '%list quality%')
  AND account_name = 'konstellationai'
ORDER BY meeting_date DESC;
```

Caveat: transcripts are konstellationai-only today. If Nick is asking about a Teknova or KAI call, check Calendar + Email in the Work base instead and say so.

### Pattern C: email triage surface

Nick: "What's in my do quadrant?"

```sql
SELECT subject, thread_summary, urgency, importance, thread_last_activity, account_name
FROM email_threads
WHERE quadrant = 'do'
  AND signal_status = 'active'
ORDER BY importance DESC, thread_last_activity DESC
LIMIT 20;
```

Caveat: `signal_status = 'active'` on 100% of rows ... lifecycle (resolved / snoozed / delegated) is not wired. You cannot distinguish "still open" from "already handled." Surface that caveat to Nick when you return results.

### Pattern D: past session pickup

Nick: "What did we work on last Tuesday?"

```sql
SELECT id, started, ended, title, summary, key_decisions, action_items, next_session_pointer
FROM agent_sessions
WHERE started::date = (CURRENT_DATE - INTERVAL '6 days')::date
  AND system_slug = 'operator-os'
ORDER BY started DESC;
```

If `next_session_pointer` is populated on the most recent row, that's where to start.

### Pattern E: artifact lookup

Nick: "Pull the canon doc on X."

```sql
SELECT id, title, path, body, updated_at
FROM canon_docs
WHERE title ILIKE '%X%'
   OR path ILIKE '%X%'
ORDER BY updated_at DESC
LIMIT 5;
```

If multiple matches, summarize titles + paths and let Nick pick.

### Pattern F: semantic search

For long / fuzzy queries, use the `recall` semantic move (defined in `practices/operator-os/CLAUDE.md`):

```
recall({
  query: "what did we decide about email triage lifecycle",
  sources: ["agent_sessions", "transcripts", "email_threads"],
  limit: 5
})
```

This dispatches a vector similarity query against `chunks` filtered by source_type, then joins back to the source rows. Returns hits with row IDs you cite in `canon_refs`.

## Citation discipline

When you call `log_conversation`, populate `canon_refs` as a newline-separated list:

```
canon_engine.transcripts:f1e2d3c4-b5a6-7890-...
canon_engine.email_threads:9a8b7c6d-...
canon_engine.agent_sessions:11223344-...
canon_engine.canon_docs:abcdef12-...
```

Do not copy content. Do not paraphrase. Cite row IDs. Future sessions resolve them live.

Asset citations (System Registry) use a parallel format on `asset_refs`:

```
registry.assets:rec9n91JItAKCKSEQ
```

## Known limitations to surface honestly

- **Transcripts: konstellationai only.** No Teknova, KAI, prospect, or personal call transcripts in canon today. If Nick asks about a call outside that scope, say so.
- **email_threads.account_name is null on 75% of rows.** Routing-by-engagement queries miss most threads. If Nick repeatedly asks for an account-scoped email view, suggest the backfill (a Roadmap item).
- **signal_status = 'active' on 100% of threads.** Lifecycle not wired. Cannot distinguish open vs handled in canon alone.
- **chunks does not include source_type='agent_session' yet.** Semantic recall across past sessions is structured-text-only (Pattern A's ILIKE) until embeddings extend.
- **canon_artifacts is empty.** Schema exists, zero rows. Don't query.
- **Ingestion pipeline provenance unverified.** Email, transcript, embedding, and canon-docs pipelines are running (data proves it) but the source code location is "PENDING" on the Registry Asset rows. Boris's problem, not yours; just know that ingestion is real but un-mapped.

## Cross-DB stitching

You cannot SQL-JOIN across the Work base (Airtable) and canon_engine (Supabase). You do logical joins by querying each and combining in code (or in your head):

- Work base Project → canon_engine transcripts/emails by `account_name` or by Project name in topics.
- Work base Task → canon_engine `agent_sessions.action_items` mentioning the Task.
- Work base Weekly Intent theme → canon_engine `agent_sessions.topics` for the week.

When citing across systems on the session row, populate both `canon_refs` (canon_engine record IDs) and `asset_refs` (Registry record IDs).

## When to escalate to Boris

Not your jurisdiction:

- Schema changes to canon_engine (new tables, new columns, new indexes)
- Ingestion pipeline issues (Gmail sync stopped, Voyage embedding failing, transcripts empty for a known meeting)
- `_ai_context` contract changes in canon_engine
- Performance issues (slow chunks queries, growing voyage_usage_log)

Report to Boris by writing the issue as a Notification in the Work base, tagging `source = "canon_engine"`. Don't fix canon_engine yourself.
