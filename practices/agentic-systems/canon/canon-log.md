## 2026-05-18 — Observable surface is truth; agent narration is hypothesis
**Surfaced during:** the Teknova L2/R5 verification work — repeatedly, an agent's summary of what it did diverged from what the surface showed (the Pfizer false positive, the raw-B vs prose review, "deployed/verified" vs actually-run).
**Primitive:** Never accept an agent's narration of its own work as the record. Verify against the observable surface (the database row, the deployed config, the run log) before asserting state. A summary describes intent, not outcome.
**Why it generalizes:** every agentic system has this gap between what the agent reports and what is true. It is the root cause of cascading error in autonomous work.
**AOS implication:** AOS must make the observable surface the system of record and treat all agent output as hypothesis pending surface verification.
**Refs:** reference_inngest_observable_architecture.md

## 2026-05-18 — "Built" is not "verified" is not "run"
**Surfaced during:** L2 v4 R5 was deployed and "ready" but never executed; the data was still old-vintage. Conflating these three states nearly authorized a destructive run on a false premise.
**Primitive:** Build, verify, and run/promote are three distinct states. An asset can be deployed and correct as source yet never have executed. State language must name which.
**Why it generalizes:** agentic work produces deploy-but-dormant artifacts constantly; collapsing the states is how phantom-progress accumulates.
**AOS implication:** AOS asset state must be an explicit lifecycle (built → verified → deployed → run/promoted), never a boolean "done."
**Refs:** feedback_build_vs_verified_language.md, feedback_build_asset_lifecycle.md

## 2026-05-18 — Three observability layers: run, process, asset
**Surfaced during:** the operator could see runs (Enrichment Runs) and process (Play Steps) but had no record of which workflows existed / which was prod — orphaned-workflow spaghetti.
**Primitive:** Agentic systems need observability at three layers — run (what executed), process (where the work is), and asset (what exists and its lifecycle state). Missing any one forces the human to micromanage.
**Why it generalizes:** the asset layer is the most commonly missing and the one whose absence most directly causes operator burnout and untrusted state.
**AOS implication:** AOS must provide all three; the asset registry is the layer most current tooling omits.

**Surfaced during:** a parallel session searched the wrong path, concluded the canonical artifact didn't exist, and nearly re-authored it from scratch — destroying provenance and desyncing a live build.

## 2026-05-18 — Governance/influence is a fourth observability layer
**Surfaced during:** a contact-sourcing workflow was over-constraining on company size / title∧department and stalling output; the operator could not see *which criteria, from which artifact at which version, were governing that AI's behavior* without reading a long analysis. He had stopped correcting the recurring size-obsession out of fatigue because the correction had nowhere durable to live.
**Primitive:** Beyond run/process/asset observability, the operator needs a **governance surface**: a small, viewable map of which artifacts govern which workflows/agents, at which version, and the specific criteria currently in force. Without it, "why did the AI do that" requires archaeology, and criteria corrections that live only in the operator's head get silently re-litigated by every new workflow and session.
**Why it generalizes:** every agentic system has criteria/policy flowing from documents into agent behavior; the influence path is invisible by default. Repeated un-pinned correction is operator burnout's root cause — same shape as the asset-registry gap, one level up (policy lineage, not asset inventory).
**AOS implication:** AOS needs an influence/governance registry (artifact → version → governed workflow/agent → in-force criteria), the policy-lineage sibling of the asset registry. Standard-practice analogues: data lineage/provenance, policy-as-code with a policy registry, requirements traceability matrix.

## 2026-05-18 — Governance is a binding/enforcement problem, not a documentation problem
**Surfaced during:** Nick's challenge that the folder/memory approach is unreliable — AI doesn't always read context, he must remember which folder to launch in, not business-scalable. This session itself proved it: a parallel session nearly re-authored the artifact despite the pin existing, because adherence depended on it choosing to consult memory.
**Primitive:** Context in a file is inert. Reliability is not a property of the document; it is a property of the binding mechanism that forces the context into the session or workflow regardless of agent discretion or human memory. Retrieval-by-agent-discretion does not compose into reliability. The reliable pattern is delivery as a runtime property, not an agent behavior: dependency injection (runtime provides context, session does not fetch it), Policy Enforcement Point not just Policy Decision Point (a gate the agent cannot bypass, not a rule it should have read), and runtime-bound configuration (the workflow consumes the governing criteria as a versioned parameter at execution — proven in our own stack: L2 ← Classification Rules, version-stamped per run).
**Why it generalizes:** every org adopting agentic AI hits this exact wall — knowledge bases / memory that the agent may or may not load. It is the central unsolved problem of applied agentic systems, and the highest-value one.
**AOS implication:** Canon Engine's product is NOT the knowledge base — it is the binding/enforcement layer. AOS governance = (automated) runtime config injection + version-stamped runs; (agent) harness injection (not retrieval) + enforcement gates + reconciliation. Content layer is necessary, weakest, and already mostly built; the delivery/enforcement layer IS the AOS.

## 2026-05-18 — Parallel agent sessions desync through shared assets unless write-ownership is assigned per asset
**Surfaced during:** four concurrent Teknova-pipeline sessions each ended cleanly with a correct self-contained handoff, but the four together were not parallel-safe — they shared a versioned artifact, two n8n workflows (where the MCP edit path wipes credentials), and one Airtable table with racing fixture cleanups. Every session was individually correct; the desync risk was structural.
**Primitive:** Concurrent stateless sessions writing shared mutable state do not compose by coordination. The reliable structure is a single-writer assignment per contended asset plus a single owner for any destructive step, gated on one reconciliation — captured in a lane map a cold session reads before writing. Handoffs that pin *state* but not *write-ownership* let two cold sessions each correctly conclude they should edit the same asset. Convergence (two lanes reaching the same decision) is not a collision — route it to one owner to reconcile, do not let both act.
**Why it generalizes:** any multi-agent system that parallelizes work over shared artifacts hits this; per-session correctness does not imply joint safety. Ownership-per-asset is the general fix, the concurrency-control sibling of the observable-surface and binding-layer primitives.
**AOS implication:** AOS must model asset write-ownership and a destructive-step gate as first-class, not rely on session discretion or operator memory to serialize parallel work; the lane map is a runtime-injected control surface, not a doc sessions are trusted to consult.

## 2026-05-18 — Structured data must be self-describing at the point of access
**Surfaced during:** moving the workflow inventory from a markdown file into an Airtable registry. Nick flagged it as a felt regression toward a schema-based SaaS mentality, with the test: "a fresh AI pointed at this base with no context would crawl it and guess what it is." The base had clean structure and zero embedded intent.
**Primitive:** Structure and context are not opposed; the error is letting either impersonate the whole. Two failure modes: the SaaS trap (structure with no embedded intent → a cold agent reverse-engineers a plausible-but-wrong model) and the inert-doc trap (intent with no structure → unbound narrative drifts, unread). The resolution is self-describing structure: the structured surface carries, at the point of access (read-first meta record, table/field descriptions that state intent not just type), the context that explains it — and that context is pinned to the structure bidirectionally so neither drifts unnoticed. A migration that moves structure but leaves its operating context behind has produced the SaaS artifact, not the agentic one.
**Why it generalizes:** every agentic system reaches for structured state as it scales; the universal failure is shipping the schema and assuming the operating model lives in someone's head. This is the binding/enforcement primitive applied to the structure↔context relationship specifically: self-description at point of access is the enforcement mechanism for structured data.
**AOS implication:** AOS's structured surfaces (registries, control planes) must ship with an in-surface read-first operating layer and bidirectionally-bound context docs as a non-skippable part of "the surface exists." Schema generation without intent embedding is incomplete. The context/folder layer is not replaced by structured state — it is promoted to the operating manual for it.

Registration is not a claim that understanding is finished; it is the act of beginning to track evolving understanding, in the open, bound to reality. ## 2026-05-18 — Intent engineering: a registry needs destination + convergence, not just state and backlog (recognized, deferred)
**Surfaced during:** Teknova's Definition Maturity read `stable` while still being built. Nick rejected "definition of done" (living systems never finish) but wanted the "aspirational destination articulated" and to know "whether we're building toward that" — the temporal context the structured DB should follow.
**Primitive:** Three layers, not two. State (what is — Assets/snapshot) and backlog (what we'll do — Roadmap) were present; **intent** was missing: what the system is becoming and whether it is converging. Intent = (1) aspirational state — a living description of the system operating as intended in steady state, not a done-date; (2) convergence — the honest read of whether current trajectory + roadmap actually close the gap, or are busy-but-drifting. Distinct from Definition Maturity, which is epistemic (how settled the *understanding* is) and orthogonal to trajectory (a definition can be stable while the build is barely started). Load-bearing insight: a roadmap proves there is work; it does NOT prove the work points at the destination — every item can complete and the system still not arrive, because the target was never pinned as what the roadmap is checked against. This is the run-vs-build family extended into a vector: not just what state, but what it converges toward and whether it is.
**Why it generalizes:** every operated system needs "are we on track" answerable from the structured surface, not from vibes; the universal miss is mistaking a backlog for a direction. Intent engineering is the discipline of making the target explicit and convergence legible.
**AOS implication:** AOS systems carry an articulated aspirational-state (context, revisable) and a convergence read; whether convergence is ever a structured field or stays an eyeballed gap vs a destination doc is deferred until reality forces it (do not anticipatorily model — emergent-systems primitive applies). Recognized, named, deliberately parked by Nick.

## 2026-05-18 — Never encode a foreseeable variant axis into the immutable identity key
**Surfaced during:** Teknova registered as `teknova-aav-enrichment`; Nick noted it will cover more than AAV and floated "rename later."
**Primitive:** "Rename later" is exactly what the identity principle forbids being cheap — the immutable join key cannot carry a foreseeable expansion axis (segment, region, product line, channel) without guaranteeing a forbidden rename when variant two appears. ## 2026-05-18 — Orchestrator removes the human from relay long before from authorization (recognized, deferred)
**Surfaced during:** Nick defined the working model — agentic-systems agent as orchestrator (plan/gates/registry/review), builders execute, he relays for now, and eventually the orchestrator spawns builder subagents so he is not the liaison.
**Primitive:** "Can the orchestrator spawn a subagent" is mechanically trivial and the wrong question. The human-in-the-loop is doing double duty: relay AND the approval gate on irreversible action (deploys, credential wipes, schema, spend). Spawn evolution is staged, not binary: safe-class subagents (research, source-reading, plan authoring/checking, dry analysis — no irreversible side effect) are spawnable autonomously now; an execute-builder that performs irreversible writes stays human-gated until (a) the observable/reconciliation layer is trustworthy so the builder's claims verify against the surface rather than being believed, and (b) per-run authorization is explicit (no-autonomous-spend and build-asset-lifecycle are not waived by a topology change). The failure mode is collapsing "spawn a subagent" into "autonomously run irreversible builds." Remove the human from relay long before from authorization.
**Why it generalizes:** every agentic org reaching for orchestrator/builder autonomy hits this; the liaison is silently also the safety interlock, and removing it for convenience removes the interlock unnoticed. The decomposition (safe-class now / irreversible gated on observability + explicit auth) is general.
**AOS implication:** AOS orchestration layer ships safe-class spawn first; autonomous execute-spawn is gated on the reconciliation/observability layer existing (same dependency the registry's own reliability has). The orchestrator/builder topology itself is eventually a registered system on the AOS line. Captured, deliberately deferred.

## 2026-05-18 — Capture wide, derive narrow; record grain decides structure
**Surfaced during:** the currency-gate investigation found the CT.gov fetch requested only 4 modules — status/dates discarded at capture. Nick generalized (esp. for paid Explorium data) and asked whether every source gets its own table.
**Primitive:** Two rules usually conflated. (1) Capture wide, derive narrow: capture the full source payload raw; structure only what is used; narrowing belongs at the curated layer, never at ingestion (architectural form of "never filter during ingestion" — raw layer ≠ curated working surface). (2) Grain decides structure, NOT "a table per source": a source producing a different-grain record than core entities (CT.gov trials: many per company, own status/dates) earns its own linked table and is a real entity, not a dump; a source enriching an existing entity at the same grain (Explorium → one company) is captured as a full raw payload on that entity, no new table until history/versioning is a real need (emergent rule). Both failure modes are real: reflexive table-per-source, and cramming a different-grain source into the core entity (the prose-only NCT id is the latter).
**Why it generalizes:** every multi-source data system faces capture-scope and where-does-it-land decisions; the universal errors are filtering at capture (unrecoverable, esp. paid) and letting "source" rather than "grain" drive schema.
**AOS implication:** AOS ingestion captures full payloads to a raw layer by default; curated surfaces are projections; new tables are grain-pulled and reality-pulled, not source-reflexive. CT.gov Trials table is a revops-engine roadmap item, deliberately NOT bolted onto the Phase 1 currency plan (sequencing: minimal safe fetch change ships now; table is separate work).

## 2026-05-18 — Selective-run is a build standard, not an ad-hoc patch
**Surfaced during:** the L2 currency "bounded 6-row smoke" was impossible — the workflow always resets all 121 rows and reclassifies the full cohort. Nick: recurring problem across engine workflows (prior sessions hand-edited node expressions to run a subset); directed a standing protocol, created and implemented now.
**Primitive:** Every RevOps-engine working table carries a `Run Selected` checkbox; every engine workflow branches at top-of-run — if any row is checked, process only those and SKIP all reset/re-blank steps; if none, normal full behavior (safe by default, zero node edits to scope a run). Checkbox not status: engine workflows reset/clear status as step one, so a status-encoded selection destroys itself before acting — a correctness constraint, not a preference. Load-bearing: the selection must gate EVERY destructive step, especially the reset; scoping only the read while still resetting all rows (the naive maxRecords patch) is strictly worse than a full run (wipes cohort, reclassifies subset, strands the rest). It is a non-skippable part of "done" for engine workflows, same family as n8n-safe-update and the registry.
**Why it generalizes:** any system with full-table/destructive workflows hits the "I just want to run these few" need; ad-hoc expression edits are error-prone and unsafe. A durable selector that gates destructive steps is the general fix.
**AOS implication:** AOS workflow scaffold includes a selective-run gate by construction; an engine workflow asset is not "reconciled" until selective-run support is verified on the surface; existing workflows flagged full-cohort-only until retrofitted.
**Refs:** sources/2026-05-18-selective-run-protocol.md, reference/revops-engine-selective-run-protocol.md, PLAN-currency-gate-phase1-deterministic-2026-05-18.md
**Source:** sources/2026-05-18-selective-run-protocol.md
**Refinement (same day, learned from a live failure):** the selection filter must be a STATIC Airtable formula (`OR({Run Selected}=1,{Verification Status}='needs_verification')`), never an n8n dynamic `={{ }}` expression — n8n's evaluator cannot run block-body JS in a filter and silently emits garbage → Airtable 422. The selection *decision* belongs in a Code node (full JS reliable); the *filter* is static. ## 2026-05-20 — Engine value is find / describe / capture / enrich; classification is commodity
**Surfaced during:** the Teknova AAV play, when the operator asked Boris how the engine determines if a company meets the criteria. Boris's first three answers leaned on derived verdict fields (`Verification Status`, then Yes/No fields next to evidence). Nick pushed each one back: a verdict is just something somebody wrote; the captured source content IS the answer. Then he stated the engine's value thesis explicitly: find the needle, describe the needle, capture evidence of the needle, build a complete dossier around the needle. Classification is easy; the value is the data.
**Primitive:** A multi-tenant RevOps engine's value is in capturing source content, not in writing labels over it. Default to evidence columns per source with provenance; treat any pre-computed verdict field (qualified, surfaced, classified, eligible) as a view-time convenience over the evidence, not the system of truth. "Anyone can classify; the value is the depth and provenance of the captured data." Two-layer target model is required: universal Target Definition per category (reusable, source-anchored, can carry industry vocabulary) + per-play Filter (tenant- and play-specific suppression and constraints). Conflating them destroys reuse and produces ambiguous verdict fields. Engine-layer vocabulary must be tenant-agnostic — industry terms live only in per-client / per-play artifacts.
**Why it generalizes:** every agentic system reaching for "intelligence" reaches first for classifiers because they're the visible deliverable. The classifier is the easy half; the data depth and provenance underneath it is where reliability lives. The engine that ships only classifiers and not evidence has built a SaaS form, not an agentic substrate. The same principle is the data-side sibling of the binding-layer and observable-surface canon entries: structure without evidence is the SaaS trap; evidence without structure is the inert-doc trap; both must be present and bound for the substrate to compose.
**AOS implication:** AOS must scaffold evidence-capture-and-enrich workflows as first-class, and treat any per-play verdict field as a derived view, not a stored truth. Engine-layer vocabulary must be tenant-agnostic by schema constraint; industry-specific terms in engine table/field names are a defect. The two-layer target model (universal definition + per-play filter) is the schema shape AOS plays inherit.
