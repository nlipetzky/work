Nick's agentic system, in plain English

Snapshot 2026-06-05. After the Deepline evaluation and the observability migration. Read this if you walked in cold and want the whole picture.

Nick runs a multi-client RevOps engine that finds the right companies and contacts for an outbound play, enriches them, classifies them, gates them, and ships them to a per-client Airtable for outreach. It is built on Supabase, Inngest, n8n, and Airtable. The work that just landed adds a way to actually see the engine running ... which fields came from where, what runs touched what records, where the duplicates are, where the gaps are.

What the engine does, end to end

The pipeline has four layers. Each one has a single job. They compose into a play.

LayerJobWhere it lives
SourcePull companies and people from the outside world. Clinical trial registries, PubMed, Explorium, conference rosters, funding feeds.Inngest functions writing to Supabase
SurfaceOne canonical schema across all clients. Companies, Contacts, Events. Plus per-client stamp columns like Teknova AAV Ready.Supabase revops-engine-dev + RevOps Surface Airtable base
MoverDumb pipe. Moves curated records from Surface to a client's base. Does no transformation.One reusable n8n workflow
ClientPer-client base with their own taxonomy, reviewer UX, and outreach surface.Per-client Airtable (e.g. Teknova Outreach base)

The rule that holds the whole thing together: tenant-agnostic at the hub, tenant-specific at the edge. The Surface never knows about clients except through stamp columns. Movement is dumb. Transformation happens upstream or downstream, never in the middle.

The data layer

One Postgres project (revops-engine-dev, project id mrmnyscurmkfppicqqhk) holds everything that matters.

  • Records: Companies (~9K), Contacts (~26K). Plus canonical aliases for both.
  • Wave + recipe machinery: Waves, recipes, wave contacts, lists, segments, plays, playbooks.
  • Activity history: entity_activity_log (11.9M rows, 90 day retention) is the universal record-of-what-happened. Plus enrichment_jobs (per provider call), enrichment_ledger (per recipe step), enrichment_runs (per wave).
  • Per-field source tracking: Both Companies and Contacts carry a field_provenance JSONB column. This is where "where did this email come from" is answered.

The Deepline lesson, in one paragraph

Deepline (a vendor evaluated this week) did not teach the engine anything new. Nick's archived RevOps system covers what Deepline does and more. What Deepline made obvious was a different thing: the projection layer over the engine. Deepline's UI shows records and runs side by side, every field carries its source, duplicates surface as a queue, gaps surface as views. Nick's engine had all the data to support this view but no UI for it, and no schema to expose it cleanly. That is the work that just landed.

What the new observability layer adds

Four ideas, vendor-agnostic, written down at practices/agentic-systems/reference/observability-projection-pattern.md.

  1. Records and runs cross-link. Every record knows which runs touched it; every run knows which records it produced. This was already true via entity_activity_log, just not surfaced. Visibility gap, not schema gap.
  2. Per-field provenance. Every enrichable column carries source, run id, captured-at, cost. The JSONB column already existed on Companies; it now exists on Contacts too.
  3. Duplicate review queue. A nightly-refreshed materialized view of probable duplicate company pairs, scored by domain and name similarity. Live as of today. 1,391 pairs surfaced. Operator marks each pair Merged, Not-a-duplicate, or Deferred in a duplicate_resolutions table.
  4. Gaps as views. Things the engine still needs to do, materialized as queryable surfaces. v_contacts_missing_email (1,958 today), v_companies_missing_modality (664 today), v_contacts_stale_employment (every contact until the employer field gets populated by enrichment).

Where in-flight work happens (new)

A staging schema was added today. When an enrichment run kicks off, results land in staging.companies_<batch_id> first, not in canonical Companies. Operator reviews via a UI, then either promotes (one SQL move) or discards (drop the staging tables). A staging_promotions ledger keeps the lineage so canonical records remember which staging batch built them.

This replaces the Deepline pattern of "work in a CSV, push when ready" with the equivalent inside Postgres. Same idea, no split truth, the projection still works.

How a client consumes the engine

A client (Teknova, for example) sees three things. None of them include the engine internals.

  • An Airtable base with curated, ready-to-act-on records (their Outreach base, app app...).
  • A periodic quality report showing what was sourced, classified, qualified, disqualified, and why.
  • An expert liaison (Hermes) routing questions and approvals to the right human at the client.

Everything else, including the 165+ functions, the per-field provenance, and the duplicate queue, is operator surface only. Clients see the artifacts, not the kitchen.

The system that surfaces it all (CRM-side)

Separate from the RevOps engine, Nick has a Canon Engine (Supabase project mzzjvoiwughcnmmqzbxv) which is his personal CRM source-of-truth. Inbound emails, meeting transcripts, and documents become typed Events. A small system called canon-crm-feed projects those Events into a separate Airtable CRM, deriving Waiting On from the event history and writing it to a new Motions table. The Motions rewire shipped on 2026-06-04. Once the Motions board runs green for a week (gate opens approximately 2026-06-12), this is the first system that will be extended with provenance writes as the pilot for the observability projection pattern.

What is still missing

  • The projection UI itself. The data exists; nothing renders it yet. Pilot plan: a Retool surface with Records, Runs, Duplicates, Gaps, and Staging Batches pages. About a day of work.
  • Contacts deduplication queue. Trigram similarity is wrong for contacts; they need exact match on LinkedIn URL and email plus normalized name plus employer. Deferred until the dedup key is agreed.
  • Row-Level Security. 36 public tables in the Supabase project have RLS disabled. The anon key can read or modify them all. Surfaced as an advisory; no action taken pending Nick's decision on a policy.
  • Promotion of the archived functions. 107 RevOps functions in archive are not yet wired into the new pattern. Adoption rolls out per-function after the canon-crm-feed pilot proves the shape.
  • Nightly refresh of the duplicate queue. Materialized view is live but unscheduled. Needs a pg_cron job (extension already installed) that runs nightly with the right session threshold.
