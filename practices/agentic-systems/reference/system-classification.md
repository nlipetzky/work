# System Classification — the five layers (aligned to the Konstellation Catalog)

The studio is itself an agentic organization, so its registry uses the SAME architecture KAI sells:
the five-layer Konstellation Catalog (`/Users/nplmini/code/work/accounts/ventures/konstellation-ai/reference/catalog.md`,
locked 2026-05-22, do not re-litigate). The deep research
(`agentic-system-definition-research-2026-06-04.md`) does NOT replace this model; it sharpens the
System/Asset boundary inside it.

## The five layers (read bottom-up)

```
Asset          atomic deliverable / building block
System         a coherent capability, operated/sold as one unit   (one outcome, one emit contract)
Cluster        sales bundle of Systems (legacy buyer lens)         (a System appears in MANY)
Constellation  integration architecture of Systems (agentic lens)  (a System has ONE home)
Trajectory     per-engagement sequence of Systems over time
```

Cluster and Constellation are NOT parallel groupings. A Cluster is how 20th-century buyers shop. A
Constellation is how the agentic company integrates.

## Definitions

**Asset** — an atomic deliverable: an n8n/automation workflow, a database or data structure, a
context-engineering spec, a surface (human interface). No standalone emit contract; the unit of cost
and reuse. (Research: this is the tool/component layer.)

**System** — a coherent capability bounded by an emit contract (Inputs / Outputs / Key Metrics +
stopping condition), delivering ONE outcome, operated/sold as one unit. The smallest independently
operable thing. Built from Assets. Has exactly ONE home Constellation; may appear in many Clusters.
(Research sharpens the boundary: bound by the outcome contract, not by steps; agency is orthogonal;
never split a System by human role/SOP — a verified failure mode.)

**Cluster** — a sales-facing bundle of Systems named in legacy buyer language ("RevOps", "Customer
Expansion"). Packaging and naming, NOT integration; the Systems in a Cluster need not share
infrastructure. A System appears in many Clusters. The Cluster layer grows as buyer perception
evolves, without restructuring anything beneath it.

**Constellation** — the integration architecture: Systems that genuinely SHARE data, infrastructure,
and operating logic, operating as one integrated org-capability. EIGHT, first-principles, fixed:
  1. Canon — knows what the business knows
  2. Compass — decides what the business does next
  3. Signal — finds opportunities outside the business
  4. Forge — builds new capabilities and content
  5. Voice — speaks and listens on the business's behalf
  6. Pulse — moves transactions
  7. Guard — protects compliance
  8. Garden — grows what exists
Expanding past eight requires first-principles proof, not preference. A System has ONE home
Constellation.

**GUARDRAIL (this is what keeps us consistent with the research, not forcing narrative against it):**
a Constellation is an INTEGRATION boundary — real shared data / infrastructure / operating logic —
not a theme. Two Systems share a Constellation because they share substrate, not because they're
"about" the same topic. This ties the constellation to the research's actual axis (context /
integration), the same axis that governs when work splits. If two Systems don't share substrate, they
are not in the same Constellation just because the name fits.

**Trajectory** — the per-engagement sequence of Systems over time plus responsibility allocation. An
engagement (a client like Teknova, a venture like KAI) HAS a Trajectory; it is NOT a Constellation.

## How to classify
1. Atomic deliverable, no standalone contract → **Asset** (belongs to one System).
2. One outcome, own emit contract, operable as a unit → **System**. Give it ONE home Constellation
   (where its data/infra/logic actually integrate) and any Clusters it sells under.
3. A buyer-named bundle of Systems → **Cluster**.
4. A first-principles org-capability of integrated Systems sharing substrate → **Constellation** (one
   of the eight).
5. A client/venture's sequenced plan of Systems → **Trajectory** (not a Constellation).

## What changed and why
The 2026-06-04 first pass invented a 3-tier model (asset/system/constellation) that diverged from the
already-locked catalog three ways: it defined Constellation as a loose "ops grouping (many
outcomes)", dropped the Cluster layer, and set cardinality backwards ("one System : many
Constellations"). The catalog is authoritative: one System : ONE home Constellation, many Clusters.
Realigned 2026-06-04. The research remains the source for the System/Asset boundary (emit-contract
test, anti-SOP rule).
